## Findings

### Finding 1

`capture_frame` can succeed in producing an `.rdc`, but still reports failure because the output file name is suffixed with `_frameNNN`.

Evidence:

- Requested path: `artifacts\renderdoc_mcp_smoke.rdc`
- Actual file created: `artifacts\renderdoc_mcp_smoke_frame189.rdc`
- Tool response: `Capture completed but file not found on disk`

Impact:

- False negative status returned to the caller
- Follow-on automation cannot reliably use the returned path

### Finding 2

Post-capture MCP analysis is currently blocked because analysis tool calls fail with `Transport closed`.

Evidence:

- `open_capture` -> `Transport closed`
- `get_capture_info` -> `Transport closed`

Impact:

- Core MCP analysis workflow is unusable after capture in the current local setup
- `get_bindings`, `get_shader`, `list_resources`, `export_render_target` could not be exercised through the MCP tool surface itself

### Finding 3

The underlying RenderDoc capture data is valid. The failure is in the MCP integration layer, not in the OpenGL sample.

Evidence:

- Sample runs successfully outside RenderDoc
- CLI successfully reads the generated `.rdc`
- CLI exports a correct render target image from the capture

## Recommended Next Steps

1. Fix `capture_frame` path reconciliation so it accepts RenderDoc's `_frameNNN.rdc` naming pattern and returns the actual created file path.
2. Investigate why the MCP server connection closes before `open_capture` and other analysis tools can run.
3. Once transport is stable, rerun the same sample and explicitly verify:
   - `open_capture`

### Advanced Findings

#### Finding A1

`list_passes` returned zero passes for the advanced capture, but `get_pass_statistics` reported two synthetic passes with valid dimensions and draw counts.

Evidence:

- `list_passes` -> `count = 0`
- `get_pass_statistics` -> `count = 2`
  - pass 1: `512x512`, `2` draws
  - pass 2: `1008x681`, `2` draws

Impact:

- Frame-structure workflows that depend on `list_passes` can under-report valid OpenGL render passes.

#### Finding A2

`get_cbuffer_contents` discovered the `SceneData` constant block correctly, but returned all-zero values for both scene draws.

Evidence:

- `list_cbuffers(stage=vs)` reported:
  - block name `SceneData`
  - byte size `80`
  - variable count `2`
- `get_cbuffer_contents(stage=vs, index=0)` at event `19`:
  - `uMvp` = all zeros
  - `uTint` = all zeros
- `get_cbuffer_contents(stage=vs, index=0)` at event `23`:
  - same all-zero result

Why this is suspicious:

- The sample visibly renders two transformed, tinted quads in the offscreen pass.
- Those draws depend on non-zero matrix and tint data written into the UBO immediately before each draw.

Impact:

- Constant-buffer inspection for OpenGL UBO-backed shaders may currently be unreliable even when reflection metadata is present.

#### Finding A3

`get_bindings` on the composite draw detected both fragment textures, but reported the same bind point for both.

Evidence:

- Event `41` fragment shader resources:
  - `uSceneTexture`
  - `uLutTexture`
- Reported bind points:
  - both shown as `bindPoint = 0`

Why this is suspicious:

- The sample binds the two samplers to texture units `0` and `1` before the composite draw.

Impact:

- OpenGL multi-texture binding inspection may be losing slot accuracy even when resource names are found.

#### Finding A4

`debug_pixel` did not produce a fragment debug trace for an overlay pixel that `pixel_history` showed was modified by the overlay draw.

Evidence:

- `pixel_history(x=201, y=170)` showed modifications from:
  - event `32` background write
  - event `56` overlay write
- `debug_pixel(eventId=56, x=201, y=170, mode=summary)` returned:
  - `No debuggable fragment at (201,170) for event 56`

Impact:

- Pixel debugging may not be consistently available for this OpenGL/blended fragment scenario.
